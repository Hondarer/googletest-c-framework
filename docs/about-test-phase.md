# テスト コードの各フェーズについての解説

テスト コードは、4 つの必須フェーズと任意の Cleanup フェーズに分けて記載します。

本テスト フレームワークには、以下のコメントからテスト エビデンスを生成する機能があります。

それぞれのフェーズの目的と具体的な内容について説明します。

- `TEST`、`TEST_F`、`TEST_P` などのテスト本体では、`{` の次の非空行を `// Arrange` とします。
- ローカル変数の宣言、モック オブジェクトの生成、テスト データや設定の準備は、すべて `// Arrange` より後に記載します。
- `[状態]` タグは、Arrange フェーズの内容をテスト エビデンスへ出力する場合に使用します。
- `[状態]` タグを付けない場合も `// Arrange` は省略しません。

## Arrange フェーズ [状態]

### 目的

このフェーズでは、テスト対象のオブジェクトや依存関係を準備し、テストを実行するための初期状態と前提条件を設定します。

### 内容

- テスト対象のオブジェクトをインスタンス化します。
- 必要なモック オブジェクトを作成します。
- テスト データを準備します。

### 例

```cpp
// Arrange
// テスト対象のクラス
MyClass myObject;

// モック オブジェクトの作成
MockDependency mockDependency;
```

## Pre-Assert フェーズ [Pre-Assert確認_*] [Pre-Assert手順]

### 目的

Act フェーズを実施する際の各モックへの呼び出しの期待を設定し、必要に応じて戻り値も制御します。

### 内容

- モック オブジェクトへの呼び出し回数や順序を確認します。
- モック オブジェクトが返すべき値を設定します。

### タグ

| タグ | 用途 |
|-----|------|
| `[Pre-Assert確認_正常系]` | 呼び出し期待の検証 (正常な動作パス) |
| `[Pre-Assert確認_異常系]` | 呼び出し期待の検証 (エラー処理パス) |
| `[Pre-Assert手順]` | 呼び出された場合の動作 |

確認と手順は役割で分離して記載します。

- 呼び出されること、呼び出されないこと、呼び出される回数、引数の内容の検証は `[Pre-Assert確認_正常系]` / `[Pre-Assert確認_異常系]` に記載します。
- 呼び出された場合に戻り値を返す、出力引数や変数を操作する動作 (`WillOnce` / `WillRepeatedly` の中身) は `[Pre-Assert手順]` に記載します。

### 例

```cpp
// モック メソッドが呼び出されたとき true を返すことを期待
EXPECT_CALL(mockDependency, SomeMethod())
    .WillOnce(Return(true)); // [Pre-Assert確認_正常系] - SomeMethod が 1 回呼ばれること。
                             // [Pre-Assert手順] - SomeMethod にて true を返却する。
```

## Act フェーズ [手順]

### 目的

このフェーズでは、Arrange フェーズで準備した状態をもとに、テスト対象のメソッドや機能を実行して戻り値を得ます。

### 内容

- テスト対象のメソッドを呼び出します。

被テスト関数の戻り値は変数で受け、Assert フェーズではその変数を検証します。被テスト関数の呼び出しを `EXPECT_*` / `ASSERT_*` の引数に直接書きません。

### 例

```cpp
// テスト対象のメソッドを呼び出す
bool result = myObject.TestedMethod(); // [手順] - TestedMethod を呼び出す。
```

## Assert フェーズ [確認_*]

### 目的

このフェーズでは、テストの結果が期待通りであることを確認し、テストの成功または失敗を判断します。

### 内容

- テスト対象のメソッドの戻り値を確認します。
- オブジェクトの状態が期待通りであることを確認します。

### タグ

| タグ | 用途 |
|-----|------|
| `[確認_正常系]` | 正常な動作パスの確認 |
| `[確認_異常系]` | エラー処理パスの確認 |

### 例

```cpp
// 戻り値が期待通りであることを確認
EXPECT_TRUE(result);  // [確認_正常系] - TestedMethod の戻り値が true であること。

// オブジェクトの状態を確認
EXPECT_EQ(myObject.GetState(), ExpectedState); // [確認_正常系] - 状態が ExpectedState であること。
```

- 関数の戻り値を確認するコメントでは、Act フェーズで呼び出した関数名を主語として記載します。
- 「戻り値が true であること。」だけでは、テスト エビデンス上でどの手順の戻り値かを判別できません。
- 同じ関数を複数回呼び出す場合は、引数、条件、呼び出し順のいずれかも記載し、対象の呼び出しを区別します。

## Cleanup フェーズ

### 目的

テストで確保または変更したリソースや状態を、Assert フェーズの完了後に元へ戻します。

### 内容

- 確保したメモリやハンドルを解放します。
- テストで作成したファイルやディレクトリを削除します。
- テストで変更したグローバル状態を復元します。

- Cleanup フェーズは任意であり、明示的な後処理がある場合だけ、その直前に `// Cleanup` を記載します。
- 明示的な後処理がない場合は、空の `// Cleanup` を記載しません。
- 終了系 API 自体の挙動を試験する呼び出しや、結果を確定するために必要な終了操作は Act フェーズへ記載します。
- fixture の `TearDown`、ヘルパー内の後処理、RAII による自動解放には、テスト本体の `// Cleanup` を記載しません。
- Cleanup フェーズにはテスト エビデンス用のタグを付けません。

### 例

```cpp
// Assert
EXPECT_TRUE(result); // [確認_正常系] - TestedMethod の戻り値が true であること。

// Cleanup
remove(outputPath);
```

4 つの必須フェーズと、必要な場合だけ Cleanup フェーズを順番に記載することで、テスト コードの役割を明確に区別できます。

## シングルフェーズ テストとマルチフェーズ テスト

テスト コードは、Arrange→Pre-Assert→Act→Assert のサイクルを 1 回だけ含む **シングルフェーズ テスト** と、同じサイクルを 1 つのテスト関数内に複数回含む **マルチフェーズ テスト** のいずれかとして記載します。状態遷移や複数ステップの手続きを 1 テストで検証する場合は、マルチフェーズ テストを使用します。

### シングルフェーズ テスト

- `// Arrange` `// Pre-Assert` `// Act` `// Assert` `// Cleanup` に番号を付けません。ここまでの節で説明した記法がこれにあたります。

### マルチフェーズ テスト

- 1 回目のサイクルの構造コメントには番号を付けません。2 回目以降のサイクルで、実際に繰り返す構造コメントにだけ `_N` を付与します (`// Arrange_2` `// Pre-Assert_2` `// Act_2` `// Assert_2`)。番号は 2, 3, 4 ... とテスト全体を通して単調増加する連番です。
- **部分反復を許容します**。4 つの構造コメントすべてが揃うサイクルの反復だけでなく、特定のフェーズだけを複数回繰り返すことができます。
    - 例: `// Arrange` → `// Pre-Assert` → `// Act` → `// Act_2` → `// Act_3` → `// Assert` のように、Act だけを複数回行い、最後に 1 回だけ Assert で確認するシナリオです。
    - この場合、`// Arrange` `// Pre-Assert` `// Assert` は 1 回しか出現しないため番号を付けません。`// Act` の 2 回目以降にだけ `_2` `_3` を付けます。
- 4 つの構造コメントは、テスト全体を通してそれぞれ 1 回以上存在することを必須とします。個々のサイクル (番号) ごとに 4 つすべてが揃っている必要はありません。
- `// Cleanup` には番号を付与しません。各サイクルの Assert 直後に、そのサイクルで確保したリソースだけを解放する書き方と、全サイクル完了後にまとめて 1 回だけ解放する書き方のどちらでも、`// Cleanup` は番号なしのまま使います。エビデンス整理ロジックは `// Cleanup` をサイクル境界の判定対象に含めず、意味のある処理としても扱いません。
- ブラケット タグ (`[状態]` `[Pre-Assert確認_*]` `[Pre-Assert手順]` `[手順]` `[確認_*]`) には番号を付けません。どのサイクルに属するかは、直近に出現した構造コメントの番号から自動的に判定されます。

### サイクル境界の判定ロジック

エビデンス整理ロジックは、`// Arrange` `// Pre-Assert` `// Act` `// Assert` のいずれかの構造コメントを検出するたびに、その番号 (省略時は 1) がそれまでの最大値を超えていれば、その番号を現在のサイクルとして更新します。超えていなければ現在のサイクルは変わりません。

例 (Act だけの部分反復):

```text
// Arrange       → 現在サイクル 1
// Pre-Assert    → 現在サイクル 1
// Act           → 現在サイクル 1 (以降のタグはサイクル 1)
// Act_2         → 現在サイクル 2 に更新 (以降のタグはサイクル 2)
// Act_3         → 現在サイクル 3 に更新 (以降のタグはサイクル 3)
// Assert        → 番号 1 は現在サイクル 3 を超えないため更新なし (タグはサイクル 3 のまま)
```

### エビデンス出力

シングルフェーズ テストでは `### 状態` / `### 手順` / `### 確認内容 (正常系:n1, 異常系:n2)` を番号なしで出力します。

マルチフェーズ テストでは、サイクルごとに次の形式を連続して出力します。

```text
### 状態_N

### 手順_N

### 確認内容_N (正常系:n1, 異常系:n2)
```

## テスト エビデンス (results.log)

`make test` を実行すると、各テスト ケースごとに results.log が生成されます。

### 出力先

```text
<テストディレクトリ>/results/<TestSuite>.<TestName>/results.log
```

**例**:

```text
app/porter/test/src/libporter/potrDisconnectPeerTest/results/potrDisconnectPeerTest.normal_health_dead/results.log
```

### ファイル構成

results.log は `----` で区切られた 3 つのブロックで構成されます。

```text
Running test: <TestSuite>.<TestName> on bin/<TestBinary>
----
<ブロック 1: テスト項目サマリ>
----
<ブロック 2: 元のテストコード>
----
<ブロック 3: gtest 実行結果>
```

#### ブロック 1: テスト項目サマリー

テスト コード内のコメント タグからテスト エビデンスが自動生成されます。

```text
## テスト項目

<TEST_F 直前の // コメント>

### 状態        ← [状態] タグから生成
### 手順        ← [手順] / [Pre-Assert手順] タグから生成
### 確認内容    ← [確認_*] / [Pre-Assert確認_*] タグから生成 (件数も集計)
```

#### ブロック 2: 元のテスト コード

テスト コードそのもの (コメント含む) が記録されます。

### プラットフォーム条件分岐

C/C++ のテスト エビデンス生成では、テスト本体内にある Linux と Windows の条件分岐を実行環境に応じて評価し、有効な分岐だけをテスト項目サマリーと元のテスト コードに出力します。

条件式には、`PLATFORM_LINUX` または `PLATFORM_WINDOWS` を `defined(...)` で単独判定する次の書式を使用します。

```cpp
#if defined(PLATFORM_LINUX)
    // Linux で出力するテスト コード
#elif defined(PLATFORM_WINDOWS)
    // Windows で出力するテスト コード
#else
    // どちらのマクロも定義されていない場合のテスト コード
#endif
```

単一のプラットフォームだけを対象とする場合は、`#if defined(PLATFORM_LINUX)` または `#if defined(PLATFORM_WINDOWS)` と `#endif` を組み合わせます。

```cpp
#if defined(PLATFORM_LINUX)
    // Linux で出力するテスト コード
#endif
```

```cpp
#if defined(PLATFORM_WINDOWS)
    // Windows で出力するテスト コード
#endif
```

対応する条件分岐は、入れ子にしても実行環境に応じて評価されます。

以下の書式はプラットフォーム条件として評価されません。

- `#ifdef` または `#ifndef` を使う書式
- `#if !defined(...)` のような否定条件
- `&&` または `||` を含む複合条件
- 条件式の後ろにコメントなどのトークンを置く書式

対応していない書式ではプラットフォームによる絞り込みを行わず、プリプロセッサ ディレクティブを含めて抽出します。

`TEST`、`TEST_F`、`TEST_P` などのテスト定義全体を囲む条件分岐は評価対象ではありません。

同名のテストをプラットフォームごとに重複定義すると各定義が連結して抽出されるため、テスト本体を一つにして、本体内の差異だけを対応する条件分岐で記載します。

#### ブロック 3: gtest 実行結果

`[ RUN ] / [ OK ] / [ FAILED ]` などの Google Test の出力が記録されます。

### 出力例

以下は `potrDisconnectPeerTest.handle_null` (異常系) の results.log の例です。

```text
Running test: potrDisconnectPeerTest.handle_null on bin/potrDisconnectPeerTest
----
## テスト項目

ハンドルが NULL の場合に POTR_ERROR を返すことの確認

### 状態

### 手順

- handle=NULL で potrDisconnectPeer を呼び出す。

### 確認内容 (異常系:2)

- ERROR ログに "handle is NULL" が含まれること。
- potrDisconnectPeer の戻り値が POTR_ERROR であること。
----
// ハンドルが NULL の場合に POTR_ERROR を返すことの確認
TEST_F(potrDisconnectPeerTest, handle_null)
{
    ...
    .Times(1); // [Pre-Assert確認_異常系] - ERROR ログに "handle is NULL" が含まれること。
    ...
    EXPECT_EQ(POTR_ERROR, rtc); // [確認_異常系] - potrDisconnectPeer の戻り値が POTR_ERROR であること。
}
----
[ RUN      ] potrDisconnectPeerTest.handle_null
[       OK ] potrDisconnectPeerTest.handle_null (0 ms)
```

### タグと出力の対応

| タグ | ブロック 1 への反映 |
|-----|-------------------|
| `[状態]` | `### 状態` セクションに箇条書きで列挙 |
| `[手順]` / `[Pre-Assert手順]` | `### 手順` セクションに列挙 |
| `[確認_*]` / `[Pre-Assert確認_*]` | `### 確認内容` セクションに列挙・件数集計 |

> **Note**: タグの直後に ` - <説明文>` を続けることで、サマリーに説明が出力されます。
> 行が `-` `*` `+` または番号付きリストで始まる説明だけが件数としてカウントされます。

> **Note**: タグはタグ文字列の完全一致で抽出されます。`[手順_異常系]` や `[Pre-Assert手順_正常系]` のような亜種はエラーにならず、サマリーから黙って欠落します。

> **Note**: マルチフェーズ テスト ([シングルフェーズ テストとマルチフェーズ テスト](#シングルフェーズ-テストとマルチフェーズ-テスト) 参照) では、`### 状態` `### 手順` `### 確認内容` の見出しに `_N` が付与され、サイクルごとに分離して出力されます。

## 確認のカテゴリ分類

### 概要

`[確認]` および `[Pre-Assert確認]` タグに `_正常系` または `_異常系` のサフィックスを付けると、確認件数をカテゴリ別に集計できます。

**サフィックスは必須** とし、サフィックスなしの `[確認]` / `[Pre-Assert確認]` は使用しません。

サマリーに「カテゴリ未指定」または区分のない件数 (`### 確認内容 (n)`) が出力された場合は修正対象です。

### カテゴリの判断基準

| カテゴリ | 目安 |
|---------|------|
| `_正常系` | 機能が正常に動作することを確認する (成功パス・通常の動作) |
| `_異常系` | エラー処理や入力不正への対応を確認する (失敗パス・防御処理) |

### 集計結果の表示

カテゴリ タグを 1 つでも使用すると、テスト結果のサマリーが分類別に表示されます。

```text
### 確認内容 (正常系:n1, 異常系:n2)
```

0 件のカテゴリは表示されません。

サフィックスなしの `[確認]` / `[Pre-Assert確認]` が混入していると、以下のいずれかの表示になります。いずれも修正対象です。

```text
### 確認内容 (n)
### 確認内容 (正常系:n1, 異常系:n2, カテゴリ未指定:n3)
```

**例**: 正常系 2 件・異常系 1 件の場合

```text
### 確認内容 (正常系:2, 異常系:1)
```

> **Note**: .NET の Theory テスト (パラメーターテスト) では、パラメーター数が自動的に乗算されて表示されます。
> 例: `### 確認内容 (正常系:2 * 3, 異常系:1 * 3)`
