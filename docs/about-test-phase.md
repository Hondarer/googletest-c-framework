# テスト コードの各フェーズについての解説

テスト コードは、4 つの必須フェーズと任意の Cleanup フェーズに分けて記載する。

本テスト フレームワークでは、以下のコメントを行うことで、テスト エビデンスを生成する機能を有している。

それぞれのフェーズの目的と具体的な内容について説明する。

- `TEST`、`TEST_F`、`TEST_P` などのテスト本体では、`{` の次の非空行を `// Arrange` とする。
- ローカル変数の宣言、モック オブジェクトの生成、テスト データや設定の準備は、すべて `// Arrange` より後に記載する。
- `[状態]` タグは、Arrange フェーズの内容をテスト エビデンスへ出力する場合に使用する。
- `[状態]` タグを付けない場合も `// Arrange` は省略しない。

## Arrange フェーズ [状態]

### 目的

テストを実行するための初期状態を設定する。  
このフェーズでは、テスト対象のオブジェクトや依存関係を準備し、テストの前提条件を整える。

### 内容

- テスト対象のオブジェクトをインスタンス化する。
- 必要なモック オブジェクトを作成する。
- テスト データを準備する。

### 例

```cpp
// Arrange
// テスト対象のクラス
MyClass myObject;

// モック オブジェクトの作成
MockDependency mockDependency;
```

## Pre-Assert フェーズ [Pre-Assert確認_*] [Pre-Assert手順]

### 目的

Act フェーズが実施される際の各モックへの呼び出しの期待を設定する。  
あわせて戻り値を制御することもある。

### 内容

- モック オブジェクトへの呼び出し回数や順序を確認する。
- モック オブジェクトが返すべき値を設定する。

### タグ

| タグ | 用途 |
|-----|------|
| `[Pre-Assert確認_正常系]` | 呼び出し期待の検証 (正常な動作パス) |
| `[Pre-Assert確認_異常系]` | 呼び出し期待の検証 (エラー処理パス) |
| `[Pre-Assert手順]` | 呼び出された場合の動作 |

確認と手順は役割で分離して記載する。

- 呼び出されること、呼び出されないこと、呼び出される回数、引数の内容の検証は `[Pre-Assert確認_正常系]` / `[Pre-Assert確認_異常系]` に記載する。
- 呼び出された場合に戻り値を返す、出力引数や変数を操作する動作 (`WillOnce` / `WillRepeatedly` の中身) は `[Pre-Assert手順]` に記載する。

### 例

```cpp
// モック メソッドが呼び出されたとき true を返すことを期待
EXPECT_CALL(mockDependency, SomeMethod())
    .WillOnce(Return(true)); // [Pre-Assert確認_正常系] - SomeMethod が 1 回呼ばれること。
                             // [Pre-Assert手順] - SomeMethod にて true を返却する。
```

## Act フェーズ [手順]

### 目的

テスト対象のメソッドや機能を実行し、戻り値を得る。  
このフェーズでは、Arrange フェーズで準備した状態をもとに、実際にテストを行う。

### 内容

- テスト対象のメソッドを呼び出す。

被テスト関数の戻り値は変数で受け、Assert フェーズではその変数を検証する。被テスト関数の呼び出しを `EXPECT_*` / `ASSERT_*` の引数に直接書かない。

### 例

```cpp
// テスト対象のメソッドを呼び出す
bool result = myObject.TestedMethod(); // [手順] - TestedMethod を呼び出す。
```

## Assert フェーズ [確認_*]

### 目的

テストの結果が期待通りであることを確認する。  
このフェーズでは、テストの成功または失敗を判断する。

### 内容

- テスト対象のメソッドの戻り値を確認する。
- オブジェクトの状態が期待通りであることを確認する。

### タグ

| タグ | 用途 |
|-----|------|
| `[確認_正常系]` | 正常な動作パスの確認 |
| `[確認_異常系]` | エラー処理パスの確認 |

### 例

```cpp
// 戻り値が期待通りであることを確認
EXPECT_TRUE(result);  // [確認_正常系] - TestedMethod の戻り値が true であること。

// オブジェクトの状態を確認
EXPECT_EQ(myObject.GetState(), ExpectedState); // [確認_正常系] - 状態が ExpectedState であること。
```

- 関数の戻り値を確認するコメントでは、Act フェーズで呼び出した関数名を主語として記載する。
- 「戻り値が true であること。」だけでは、テスト エビデンス上でどの手順の戻り値かを判別できない。
- 同じ関数を複数回呼び出す場合は、引数、条件、呼び出し順のいずれかも記載し、対象の呼び出しを区別する。

## Cleanup フェーズ

### 目的

テストで確保または変更したリソースや状態を、Assert フェーズの完了後に元へ戻す。

### 内容

- 確保したメモリやハンドルを解放する。
- テストで作成したファイルやディレクトリを削除する。
- テストで変更したグローバル状態を復元する。

- Cleanup フェーズは任意であり、明示的な後処理がある場合だけ、その直前に `// Cleanup` を記載する。
- 明示的な後処理がない場合は、空の `// Cleanup` を記載しない。
- 終了系 API 自体の挙動を試験する呼び出しや、結果を確定するために必要な終了操作は Act フェーズへ記載する。
- fixture の `TearDown`、ヘルパー内の後処理、RAII による自動解放には、テスト本体の `// Cleanup` を記載しない。
- Cleanup フェーズにはテスト エビデンス用のタグを付けない。

### 例

```cpp
// Assert
EXPECT_TRUE(result); // [確認_正常系] - TestedMethod の戻り値が true であること。

// Cleanup
remove(outputPath);
```

4 つの必須フェーズと、必要な場合だけ Cleanup フェーズを順番に記載することで、テスト コードの役割を明確に区別できる。

## テスト エビデンス (results.log)

`make test` を実行すると、各テスト ケースごとに results.log が生成される。

### 出力先

```text
<テストディレクトリ>/results/<TestSuite>.<TestName>/results.log
```

**例**:

```text
app/porter/test/src/libporter/potrDisconnectPeerTest/results/potrDisconnectPeerTest.normal_health_dead/results.log
```

### ファイル構成

results.log は `----` で区切られた 3 つのブロックで構成される。

```text
Running test: <TestSuite>.<TestName> on bin/<TestBinary>
----
<ブロック 1: テスト項目サマリ>
----
<ブロック 2: 元のテストコード>
----
<ブロック 3: gtest 実行結果>
```

#### ブロック 1: テスト項目サマリー

テスト コード内のコメント タグから自動生成されるテスト エビデンス。

```text
## テスト項目

<TEST_F 直前の // コメント>

### 状態        ← [状態] タグから生成
### 手順        ← [手順] / [Pre-Assert手順] タグから生成
### 確認内容    ← [確認_*] / [Pre-Assert確認_*] タグから生成 (件数も集計)
```

#### ブロック 2: 元のテスト コード

テスト コードそのもの (コメント含む) が記録される。

#### ブロック 3: gtest 実行結果

`[ RUN ] / [ OK ] / [ FAILED ]` などの Google Test の出力が記録される。

### 出力例

以下は `potrDisconnectPeerTest.handle_null` (異常系) の results.log の例。

```text
Running test: potrDisconnectPeerTest.handle_null on bin/potrDisconnectPeerTest
----
## テスト項目

ハンドルが NULL の場合に POTR_ERROR を返すことの確認

### 状態

### 手順

- handle=NULL で potrDisconnectPeer を呼び出す。

### 確認内容 (異常系:2)

- ERROR ログに "handle is NULL" が含まれること。
- potrDisconnectPeer の戻り値が POTR_ERROR であること。
----
// ハンドルが NULL の場合に POTR_ERROR を返すことの確認
TEST_F(potrDisconnectPeerTest, handle_null)
{
    ...
    .Times(1); // [Pre-Assert確認_異常系] - ERROR ログに "handle is NULL" が含まれること。
    ...
    EXPECT_EQ(POTR_ERROR, rtc); // [確認_異常系] - potrDisconnectPeer の戻り値が POTR_ERROR であること。
}
----
[ RUN      ] potrDisconnectPeerTest.handle_null
[       OK ] potrDisconnectPeerTest.handle_null (0 ms)
```

### タグと出力の対応

| タグ | ブロック 1 への反映 |
|-----|-------------------|
| `[状態]` | `### 状態` セクションに箇条書きで列挙 |
| `[手順]` / `[Pre-Assert手順]` | `### 手順` セクションに列挙 |
| `[確認_*]` / `[Pre-Assert確認_*]` | `### 確認内容` セクションに列挙・件数集計 |

> **Note**: タグの直後に ` - <説明文>` を続けることで、サマリーに説明が出力される。
> 行が `-` `*` `+` または番号付きリストで始まる説明だけが件数としてカウントされる。

> **Note**: タグはタグ文字列の完全一致で抽出される。`[手順_異常系]` や `[Pre-Assert手順_正常系]` のような亜種はエラーにならず、サマリーから黙って欠落する。

## 確認のカテゴリ分類

### 概要

`[確認]` および `[Pre-Assert確認]` タグには、`_正常系` または `_異常系` のサフィックスを付けることで、  
確認件数をカテゴリ別に集計できる。**サフィックスは必須** とし、サフィックスなしの `[確認]` / `[Pre-Assert確認]` は使用しない。  
サマリーに「カテゴリ未指定」または区分のない件数 (`### 確認内容 (n)`) が出力された場合は修正対象である。

### カテゴリの判断基準

| カテゴリ | 目安 |
|---------|------|
| `_正常系` | 機能が正常に動作することを確認する (成功パス・通常の動作) |
| `_異常系` | エラー処理や入力不正への対応を確認する (失敗パス・防御処理) |

### 集計結果の表示

カテゴリ タグを 1 つでも使用すると、テスト結果のサマリーが分類別に表示される。

```text
### 確認内容 (正常系:n1, 異常系:n2)
```

0 件のカテゴリは表示されない。

サフィックスなしの `[確認]` / `[Pre-Assert確認]` が混入していると、以下のいずれかの表示になる。いずれも修正対象である。

```text
### 確認内容 (n)
### 確認内容 (正常系:n1, 異常系:n2, カテゴリ未指定:n3)
```

**例**: 正常系 2 件・異常系 1 件の場合

```text
### 確認内容 (正常系:2, 異常系:1)
```

> **Note**: .NET の Theory テスト (パラメーターテスト) では、パラメーター数が自動的に乗算されて表示される。
> 例: `### 確認内容 (正常系:2 * 3, 異常系:1 * 3)`
